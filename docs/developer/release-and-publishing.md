# Release And Publishing

This project is prepared for two public delivery channels:

- GitHub Pages for documentation and marketing content from `docs/`
- Visual Studio Marketplace for the final VS Code extension release

## GitHub Pages

The repository includes a Pages workflow at `.github/workflows/pages.yml`.

### What it does

- validates the docs site on pull requests
- publishes the `docs/` folder on pushes to `main`
- deploys through the GitHub Pages Actions flow

### Repository setup required

1. Open the repository settings on GitHub.
2. Go to `Pages`.
3. Set the source to `GitHub Actions`.
4. Keep the published site rooted in the `docs/` content generated by the workflow.

### Content expectations

- `docs/index.md` is the site landing page
- `docs/user/` is the user-facing product and usage area
- `docs/developer/` is the contributor and engineering area
- `docs/assets/` stores screenshots and documentation images

## VS Code Marketplace Release

The repository includes a release workflow at `.github/workflows/release-extension.yml`.

### What it does

- checks out the code
- locates the extension manifest from either the repo root or `extension/`
- installs dependencies
- runs lint, tests, and build scripts when present
- packages a `.vsix`
- attaches the `.vsix` to the GitHub release
- publishes to the Visual Studio Marketplace when `VSCE_PAT` is configured

### Required repository secret

- `VSCE_PAT`

This secret must contain a Visual Studio Marketplace Personal Access Token with permission to publish the extension under the intended publisher.

### Required extension manifest fields

Before a final release, the extension `package.json` must include at least:

- `name`
- `publisher`
- `version`
- `engines.vscode`

Recommended additional metadata for a strong public listing:

- `displayName`
- `description`
- `categories`
- `keywords`
- `icon`
- `repository`
- `license`
- `homepage`
- `bugs`

### Release tag rule

The release workflow expects the GitHub release tag to match the extension version:

- `v1.2.3`
- or `1.2.3`

That must match `package.json#version`.

### Final release flow

1. Ensure the extension builds, tests, and packages successfully in CI.
2. Ensure the extension manifest and Marketplace assets are complete.
3. Create a release tag that matches the extension version.
4. Publish a GitHub Release.
5. The workflow packages the extension, uploads the `.vsix`, and publishes to Marketplace if `VSCE_PAT` is present.

## Marketplace Availability Notes

Publishing to the Visual Studio Marketplace is the step that makes the extension discoverable to regular VS Code users from the Extensions view.

Current constraints:

- the extension code and manifest do not exist yet in this repository
- the Marketplace publisher and token must exist before public release
- extensions using proposed APIs cannot be published publicly

## Recommended Future Additions

- publish release notes from the changelog into the GitHub release body
- add marketplace badge and install links to the Pages landing page
- optionally publish to Open VSX for broader ecosystem reach outside Microsoft-distributed VS Code
